## 2. 技術選定（候補比較と理由）

### フロント＋サーバー: **Next.js (App Router) + TypeScript** ✅
| 候補 | 長所 | 短所 | 判定 |
|------|------|------|------|
| **Next.js + TS** | 画面とサーバー処理を1つで書ける／Vercelと相性最良／情報量豊富／TypeScriptで型安全（バグを事前に防ぐ） | 学習コストはやや高い | **採用** |
| Remix / SvelteKit | 軽量・高速 | Vercel/Supabase連携の事例がNext.jsより少ない | 不採用 |
| SPA (React単体) + 別API | 構成が分離して明快 | サーバー処理を別途用意する手間／SEOに弱い | 不採用 |

> 一覧・詳細は **Server Components** でサーバー側描画 → 表示が速くSEOに強い（外部からも作品が見つかる）。

### データ基盤: **Supabase** ✅
| 候補 | 長所 | 短所 | 判定 |
|------|------|------|------|
| **Supabase** | PostgreSQL＋認証＋cron(pg_cron)＋ストレージ＋RLS（行レベル権限）を**1つで**提供／無料枠／トークン保管に使える | 高度なカスタムは制約あり | **採用** |
| Firebase | リアルタイムに強い | NoSQLで「作品⇔放送予定」等のリレーショナルな整理がしづらい | 不採用 |
| 自前PostgreSQL + 認証自作 | 自由度最大 | 構築・運用・セキュリティ負担が大きい | 身内規模には過剰 |

> 「作品・話・放送予定・キャスト」のような **関係性のあるデータ** はリレーショナルDB(PostgreSQL)が最適。Supabaseは認証とcronまで含むため、本要件にほぼ過不足なく合う。

### 公開先: **Vercel** ✅
- Next.jsの開発元が運営。デプロイ（公開）がGit連携で自動。Vercel Cronも使える。

### 認証/連携: **Google OAuth + Google Calendar API** ✅（要件で確定）

### データソース: **Annict（メタ） + しょぼいカレンダー（放送時刻）** ✅
- 単独では要件を満たせない。役割分担が最適（[00](00_README_設計ドキュメント目次.md) 参照）。

## 3. 拡張に強くするための設計原則

### 原則① データ取得を1層に閉じ込める（アダプタ層）
```
[画面/API] → [自前DB] ← [取り込み層(adapters)] ← Annict / しょぼい / (将来)他ソース
```
- 外部APIに触るコードを `adapters/annict.ts` `adapters/syoboi.ts` のように分離。
- 仕様変更・ソース追加・差し替えがこの層だけで済む（画面側は影響を受けない）。

### 原則② 「名前は文字列、IDは後で」方式で将来の正規化に備える
- MVPでは声優・スタッフ・会社を **文字列** で保持しつつ、`person_id` / `company_id` の **空きカラムを用意**。
- 将来「声優ページ」を作るとき、文字列を `persons` テーブルへ正規化しIDで紐付け → ページ追加が容易（[04](04_DB設計.md)）。

### 原則③ 通知などの「派生機能」はイベント駆動で足す
- 放送時刻変更・新規追加を **同期ジョブが検知** → 通知モジュール（メール/Discord/Slack）へ渡す、という拡張点を最初から想定。
- MVPでは通知モジュールは空でよい（フックだけ用意）。

### 原則④ 公開データと個人データを明確に分離（RLS）
- 閲覧系（works等）は誰でも読める。個人系（subscriptions/トークン）は本人のみ。最初からこの線引きをしておくと、機能追加時に権限事故が起きにくい。

## 4. 将来拡張への備え（候補機能 → 実現方法）

| 将来機能 | 既存設計のどこを使うか | 追加で必要なもの |
|----------|------------------------|------------------|
| 声優ページ / 出演作一覧 | `persons` テーブル＋`work_casts.person_id` | 正規化処理・声優詳細画面 |
| スタッフページ | `persons`＋`work_staff.person_id` | 同上 |
| 制作会社ページ / 制作作品一覧 | `companies`＋`works.company_id` | 会社詳細画面 |
| シーズン別ランキング | `works`＋（閲覧数/お気に入り数の集計表） | 集計テーブル・ランキングAPI |
| お気に入り登録 | `app_users`＋新規`favorites`表 | お気に入りAPI/UI |
| 社内おすすめ共有 | `app_users`＋新規`recommendations`表 | 投稿/閲覧UI |
| 放送延期・時間変更通知 | 同期ジョブの「変更検知」 | 通知モジュール |
| Discord / Slack 通知 | 同上のフック | Webhook連携設定 |

> どれも **テーブル追加 + 画面追加** で載る構造。MVPのコア（works/programs/subscriptions/calendar_events）は変更不要。

## 5. ディレクトリ構成イメージ（実装時の目安）
```
/app                  画面 (Next.js App Router)
  /(public)/...        一覧・詳細（ログイン不要）
  /api/...             アプリ内API
  /api/internal/...    cron用 内部API
/lib
  /adapters            annict.ts / syoboi.ts / google.ts  ← 外部API層
  /db                  クエリ・型
  /sync                ingest.ts / syncCalendars.ts        ← 自動更新ロジック
  /auth                oauth.ts / tokenCrypto.ts           ← トークン暗号化
/components            UI部品（作品カード等）
```
