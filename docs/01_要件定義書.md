# 01. 要件定義書

## 1. このアプリは何か（サービスコンセプト）

放送中・放送予定のアニメ情報を **見やすく整理して一覧表示** し、気になった作品だけを **Googleカレンダーへ登録** できるWebアプリ。

- Annictのような「作品を探す・情報を見る」体験を提供する。
- ただし目的は **視聴記録ではない**。「放送予定を逃さないためのカレンダー登録」が主目的。
- 利用者は **身内・会社のメンバー**（小規模・クローズド寄り）。

### 一言でいうと
> 「次に何が放送されるか」を一覧で見て、見たい作品をワンタップで自分（や会社共有）のGoogleカレンダーに入れられるサイト。

## 2. 利用者（想定ユーザー）

| 区分 | 説明 | ログイン |
|------|------|----------|
| 閲覧ユーザー | アニメ情報を見るだけの人。大多数。 | **不要** |
| 登録ユーザー | カレンダーに追加したい人。 | **「追加」ボタンを押した時だけ** Googleログイン |

- ログインを必須にしない。「見るだけなら誰でも、登録するときだけGoogle認証」が基本方針。

## 3. スコープ（やること / やらないこと）

### ✅ やること（このプロジェクトの対象）
- ログイン不要のアニメ情報閲覧（一覧・詳細）
- 今シーズン / 来シーズン / 放送中 / 放送予定 の一覧表示
- 作品詳細（基本情報・エピソード・キャスト・スタッフ）
- 「Googleカレンダーへ追加」ボタン → Googleログイン → 登録先カレンダー選択 → 登録
- 登録した作品の放送予定を **サーバー側で定期取得し自動でカレンダーに追加**
- 同一放送予定の **重複登録防止**

### ❌ やらないこと（今回は対象外。将来拡張で検討）
- 視聴記録・レビュー・評価機能
- 声優ページ / スタッフページ / 制作会社ページ（※DB設計だけは拡張可能にしておく）
- ランキング / お気に入り / 社内おすすめ共有
- 放送延期・時間変更の通知 / Discord・Slack通知
- 一般公開・大規模ユーザー対応（まずは身内規模で運用）

> 将来候補は [10_アーキテクチャ設計.md](10_アーキテクチャ設計.md#将来拡張への備え) に「どう拡張するか」を記載。

## 4. 機能要件

### 4.1 アニメ一覧
- 区分タブ: **今シーズン / 来シーズン / 放送中 / 放送予定**
- 各作品カードに表示: **タイトル / キービジュアル / 放送時期 / 放送状況 / ジャンル（取得できれば）**
- カードをクリック → 作品詳細へ

### 4.2 作品詳細
- **基本情報**: タイトル / あらすじ / 公式サイト / 放送情報（放送局・開始時期など）
- **エピソード**: 話数 / サブタイトル
- **キャスト**: `キャラ名：声優名`（例: フリーレン：種﨑敦美）
- **スタッフ**: 役割と担当者（監督 / シリーズ構成 / キャラクターデザイン / 音楽 / アニメーション制作 など）
- **「Googleカレンダーへ追加」ボタン**

### 4.3 Googleカレンダー連携
- 「追加」ボタン押下時のみGoogleログイン（OAuth）
- ログイン後、**ユーザーがアクセス可能なカレンダー一覧**を取得（個人 / 会社共有 / プロジェクト用 など）
- その中から **登録先を選択**
- 登録形式:
  - 作品単位: `【アニメ】作品名`
  - 話単位: `【アニメ】作品名 第○話`
- 説明欄（可能な範囲で）: サブタイトル / 放送局 / 作品URL

### 4.4 自動更新（重要要件）
- ユーザーが登録した後は、**PCを起動していなくても**定期的に放送予定を取得し、Googleカレンダーへ追加する。
- → これは「サーバー側で動く定期処理」が必須。詳細は [08_自動更新方式.md](08_自動更新方式.md)。

### 4.5 重複防止
- 同じ放送予定が複数回カレンダーに登録されないようにする。
- → 「どの予定を、どのカレンダーに、すでに登録したか」を記録する台帳テーブルで実現。詳細は [07](07_GoogleカレンダーAPI連携設計.md)・[04](04_DB設計.md)。

## 5. 非機能要件（性能・運用・セキュリティなど）

| 項目 | 方針 |
|------|------|
| 規模 | 身内・会社向け。同時利用者は数〜数十人想定。過剰な性能設計はしない。 |
| 表示速度 | 一覧はアプリ側DBにキャッシュ済みデータを表示（外部APIを毎回叩かない）。 |
| 外部API負荷 | Annict / しょぼいカレンダーへは **定期バッチでまとめて取得** し、API利用規約・レート制限を守る。 |
| セキュリティ | Googleの**リフレッシュトークン**（後述）はサーバーで暗号化保存。閲覧は認証不要だが、登録・自動更新は本人のトークンでのみ実行。 |
| 個人情報 | 保持するのは Googleアカウントのメール・必要最小限のトークンのみ。 |
| 可用性 | 自動更新が一時的に失敗しても次回実行でリカバリできる設計（再実行に強い=冪等）。 |
| コスト | Supabase / Vercel の無料〜低額枠で収まる規模を想定。 |

## 6. 外部依存とリスク

| リスク | 内容 | 対策 |
|--------|------|------|
| データソース紐付け | Annict と しょぼいカレンダー の作品突合が完全自動では難しい場合がある | MVPは今シーズンのみ小規模で検証。突合できない作品は手動補正できる設計にする |
| 外部API仕様変更・停止 | Annictの放送予定ページは既に廃止。今後も変更リスク | データ取得層を1箇所に分離し、差し替え可能にする（[10](10_アーキテクチャ設計.md)） |
| Google API利用制限 | OAuth未審査アプリは利用者上限あり（テストユーザー扱い） | 身内利用なら問題になりにくいが、運用前に確認（[06](06_GoogleOAuth設計.md)） |
| 著作権・画像利用 | キービジュアル等の出典・利用範囲 | 出典元のAPI規約に従い、画像は直リンク/規約準拠で扱う |

## 7. 用語ミニ辞典（非エンジニア向け）

| 用語 | かんたんな意味 |
|------|----------------|
| API | 他のサービスからデータをもらうための「窓口」。AnnictやGoogleにはこの窓口がある。 |
| OAuth | 「このアプリにGoogleカレンダーへの書き込みを許可しますか？」の同意の仕組み。パスワードは渡さない。 |
| リフレッシュトークン | 一度許可をもらえば、PCを切っていてもアプリがGoogleに代わってアクセスし続けられる「合鍵」。要厳重保管。 |
| バッチ / cron | 決まった時刻に自動で動くサーバー処理。「毎日6時に放送予定を取得」など。 |
| 冪等（べきとう） | 同じ処理を何回やっても結果が変わらない性質。重複防止の鍵。 |
